# 赛题对齐与后续开发路线

本文档记录 SparkWeave 面向“基于大模型的个性化资源生成与学习多智能体系统开发”赛题的功能对齐情况和后续开发计划。后续开发优先围绕这里的五条主线推进，避免功能继续发散。

## 赛题要求拆解

| 赛题要求 | SparkWeave 已有基础 | 当前判断 |
| --- | --- | --- |
| 对话式学习画像自主构建 | Chat 会话、Memory、Notebook、导学画像、题目本、会话历史、证据账本、用户校准 | 已进入统一画像中心阶段，重点转为演示可解释性和持续更新 |
| 多智能体协同的资源生成 | Chat 协调、Deep Solve、Deep Question、Deep Research、Visualize、Math Animator、SparkBot、资源卡协作链路 | 项目强项，前端已开始用轻量摘要展示协作链路，避免暴露运行日志 |
| 个性化学习路径规划和资源推送 | Guide V2、课程模板、当前任务、前测补基、资源生成、任务证据、提交回写 | 主干已改为懒人式当前任务流程，后续继续打磨课程 Demo 和稳定产物 |
| 智能辅导 | RAG、文字答疑、图解、Manim 短视频、题目生成、视觉输入、OCR、GeoGebra | 能力较完整，演示稳定性和多模态产物呈现需要重点保障 |
| 学习效果评估 | 题目本、答题结果、导学任务提交、Notebook 复盘、反思记录、学习报告、画像证据变化 | 已具备闭环雏形，下一步重点是把评估结果讲得更像“学习建议”而不是报表 |

## 总体开发目标

项目后续不再单纯堆功能，而是围绕一个完整学习闭环收束：

```text
确定学习目标
  -> 对话式画像构建
  -> 前测诊断
  -> 个性化路径规划
  -> 当前任务推进
  -> 多智能体生成资源
  -> 交互练习与答疑
  -> 提交反思和证据
  -> 学习效果评估
  -> 动态调整画像、路径和资源推荐
```

前端体验原则：

- 用户每次只需要知道“现在该做什么”。
- 页面少平铺、多分页，减少折叠区和复杂工具面板。
- 面向普通学习者，而不是面向开发者或调试者。
- 多智能体协作要可见，但不能变成噪音。
- 资源、练习、反馈和报告都要形成闭环。

## 2026-05-01 当前实现状态

- 画像中心已经从“标签展示”升级为“证据驱动”：对话、导学、练习、反思和笔记都可以沉淀为画像证据，并通过 Memory 页面做面向用户的解释。
- 导学页面已经按“先做这一件事”收束，不再把资料、报告、推荐和调试信息平铺在首屏。
- 导学资源替代选择已经移到“换一种材料”子页：主任务卡只保留系统推荐资源和提交入口，替代选择降级为小入口，图解、练习、短视频、精选视频等不再平铺干扰主流程。
- 导学资源已经改为任务内产物：图解、自生成短视频、精选公开视频、练习等围绕当前任务生成，并带有“生成依据”和轻量“协作链路”。
- 导学材料卡片的协作链路升级为“智能体接力”路线：用“画像 -> 专项资源智能体 -> 评估”三步表达资源为什么生成、由谁生成、如何回到学习闭环，避免暴露调试日志。
- 练习已经朝交互式提交推进：题目不只是展示文本，而是服务于提交、反馈、画像回写和下一步建议。
- 导学交互练习新增整组结果预览：逐题提交后会直接显示本组正确数和下一步提示，再由用户一键回写学习路径。
- 学习效果报告开始输出“学习处方”：把分数、错因、画像信号压缩成一个首要行动和两个备选动作，并能直接触发图解、练习、复测或课程产出包入口。
- 学习处方新增三步预告：先说明现在做什么，再说明完成后提交什么，最后说明系统如何回写画像和调整路线，让学习效果评估更像真实辅导闭环。
- 学习处方触发生成后，会在报告下方显示“处方产物”，避免用户再去任务页里找结果。
- 学习报告新增“演示就绪度”：轻量检查画像证据、多模态资源、练习闭环、学习处方和可展示产物是否成链，帮助录制 7 分钟比赛演示前快速补齐缺口。
- 课程产出包新增“7 分钟演示路线”：按画像、路线、资源、练习、报告、产出包顺序给出录屏片段、赛题映射和兜底材料。
- 课程产出包新增“录屏兜底包”：固定学习者画像、稳定素材、检查项和兜底话术，降低现场模型波动对演示的影响。
- 内置“机器学习基础”课程新增稳定 Demo 样例：固定画像、T1/T4/T6 任务链、资源提示词和示例反思，便于复现演示。
- 导学首页新增“开始稳定演示”入口：一键创建机器学习课程 Demo 路线，减少录屏现场配置步骤。
- 稳定 Demo 任务页新增快捷动作：命中 T4/T6 等样例任务时直接显示稳定资源提示词，提交页可一键带入示例分数和反思。
- 稳定 Demo 反馈页新增“演示收尾”入口：提交学习证据后，只显示就绪度摘要、最小补齐建议，以及路线和产出包入口，让录屏闭环更顺。
- 稳定 Demo 录屏路径新增 Playwright 黑盒保护：从一键创建、稳定提示词生成、示例反馈带入、演示收尾到课程产出包，确保后续 UI 简化不会破坏主线。
- 稳定 Demo 主流程新增“录屏下一步”提示器：只提示当前该点什么和该讲哪一句，不展开完整脚本，保持懒人式体验。
- 课程产出包新增“录屏检查”单卡片：把演示路线、兜底包和稳定样例压缩成一个入口，打开后先看三步脚本和一条兜底动作。
- 课程产出包新增“画像驱动产出”说明：把推进风格、薄弱点、资源/练习证据写入产出包和 Markdown，录屏时能直接说明画像如何影响课程路线与作品组织。
- 课程产出包新增“赛题五项对齐”卡片：把画像、路径、资源、智能辅导和学习效果评估映射到当前课程证据，并给出下一步录屏动作，方便答辩时直接对齐评分点。
- 稳定 Demo 样例新增结构化 `sample_artifacts`：课程模板可以提前写入图解节点、预置练习、视频分镜、公式重点和学习处方，录屏时即使现场生成变慢，也能展示稳定素材和讲述要点。
- 课程产出包新增“答辩问答预案”：自动整理评委可能追问的画像、多智能体、个性化路线、学习评估和稳定性问题，并指向对应演示页面。
- 课程产出包页面新增“下载 Markdown”按钮：比赛材料可以直接导出为 `.md` 文件，便于整理 PPT、提交文档和录屏讲稿，不必只保存在 Notebook 内部。
- 学习报告页面新增“下载 Markdown”按钮：学习效果评估、学习处方、错因复测和演示就绪度可以独立导出，方便单独整理加分项材料。
- 文档中心新增“比赛 7 分钟演示 Runbook”：按分钟拆分页面动作、讲法、赛题映射、兜底策略和答辩优先回答，降低正式录屏组织成本。
- 课程模板能力继续增强：`data/course_templates/*.json` 现在可以直接生成确定性导学路线，不必依赖现场大模型规划；已新增“智能机器人与 ROS 基础”“高等数学极限与导数”“大模型教育智能体系统开发”完整课程模板，分别适合展示项目式导学、公式/动画型多模态辅导和赛题主线课程闭环。
- 导学首页新增完整课程快选：除“一键稳定演示”外，用户还能直接点选“智能机器人与 ROS 基础”等课程卡片，系统自动填好目标、时间和偏好，减少下拉配置，让完整高校课程 Demo 更容易启动。
- 外部课程模板的演示种子已完成前端归一化：`task_chain` 可以使用任务 ID 数组，`resource_prompts` 可以使用任务 ID 到提示词的映射；导学页会自动还原为稳定任务快捷入口，保证 ROS/高数等课程也能一键生成图解、练习或短视频素材。
- 导学新增 `external_video` 精选视频资源：复用联网搜索检索公开视频，按任务、画像弱点和学习偏好筛选少量可观看视频卡片，补足“系统自己生成 + 网络优质资源推荐”的实用闭环。
- 对话页也可直接触发 `external_video_search`：用户无需切换页面，只要提出找公开视频、公开视频课或讲解视频，协调智能体会把任务交给视频检索智能体并以卡片形式返回。
- 对话协调智能体的视频意图识别继续细化：例如“找一个视频讲解 / 推荐 B 站公开课 / 网课视频”会走公开视频检索，而“生成一个视频讲解 / Manim 动画”仍走数学动画智能体，减少错误唤醒。
- 对话页空状态已接入统一画像的 `next_action`：首屏不再只提示“可以提问”，而是给出“今天先做这一步”，并提供进入导学、直接开始、生成练习、找公开视频和画图解的一键动作。
- 对话页“按画像继续”入口改为真正的懒人指令：前端只发送“继续学习”，由对话协调智能体读取画像并改写成具体学习任务，用户不需要看到或理解内部提示词。
- 对话协调智能体已接入画像提示：在唤醒题目、图解、动画、研究路径和公开视频检索智能体时，会携带当前薄弱点、偏好、时间预算和下一步任务，让资源生成更贴合学习者当下状态。
- 注入给大模型的画像上下文继续压缩增强：除当前目标、薄弱点和偏好外，新增 `preferred_resource` 和 `next_action_prompt`，让协调智能体能更稳定地选择图解、交互练习、短视频或精选公开视频。
- 注入给大模型的画像上下文继续补齐“推进方式”：除资源偏好外，模型现在还能看到练习驱动型、概念澄清型、快速串联型等策略提示，避免个性化只停留在前端展示。
- 对话协调智能体新增画像驱动兜底调度：用户只说“继续学习 / 下一步 / 我该做什么”时，系统会读取画像里的首选资源和下一步提示，主动唤醒公开视频、出题、图解或动画智能体，而不是停在普通聊天。
- 对话页的多智能体协作展示已经从原始事件流改成用户可读的“协作明细”和“协作状态”：保留画像参与、角色接力和关键动作，不再暴露 `stage_start`、`thinking` 等调试事件。
- 对话页协作面板新增“画像触发”提示：当协调智能体把“继续学习”这类泛化指令改写成具体学习任务时，前端会展示改写后的任务，让用户看懂画像如何驱动智能体接力。
- 对话页新增“接力路线”可视化：把画像、检索、生成、校验、讲解等角色压缩成一条横向路线，用户先看懂谁在接力，再决定是否展开协作明细。
- 对话页继续去开发者化：首屏、历史会话和当前状态都使用“即时答疑 / 学习会话 / 当前回答”等用户可读表达，不再直接暴露 `chat`、会话 ID 或轮次 ID。
- 对话页完成态轨迹继续收束：如果最终回答已经生成，即使后端随后还有 `writing/progress` 这类尾部事件，前端也只展示最终回答、工具、资料和关键观察，避免出现“答案已完成但仍显示进行中”的错觉。
- 图解、动画和精选视频结果已返回画像生成依据：前端用“按你的画像生成”小卡片解释当前资源参考了哪些薄弱点、偏好、时间预算或下一步任务，避免用户只看到产物却不知道个性化从何而来。
- 学习画像页新增一键确认：用户可以直接告诉系统“这个判断准确”，作为显式校准证据写回画像，减少只有纠错入口的压力。
- 学习画像概览新增“推进方式”卡片：把练习驱动型、概念澄清型、快速串联型等风格判断，以及最近趋势压缩成用户能看懂的一张卡，避免画像变成开发者数据面板。
- 精选视频结果从搜索链接升级为学习卡片：展示建议观看顺序、看完后的下一步动作和轻量筛选链路，避免用户陷入无目的刷视频。
- 精选视频结果继续升级为“学习计划卡”：后端返回三步观看计划和看完后的反思问题，前端会写入画像证据元数据，让公开视频也进入“看 -> 反思 -> 画像更新 -> 下一步推荐”的闭环。
- 学习画像页现在能展示公开视频证据里的观看计划和反思问题：用户不只看到“系统记录了我看过视频”，还能看到这条证据下一次如何影响资源偏好和学习处方。
- 精选视频检索已利用搜索结果里的结构化信息：会读取缩略图、频道、时长等字段，并结合画像里的时间预算优先推荐短讲解，避免把 10 分钟补基任务推成 45 分钟公开课。
- 精选视频检索的真实搜索兼容性继续增强：除统一 `search_results/citations` 外，也能识别 `results/organic/data/items/videos` 等常见返回形态，并兼容 YouTube live、Bilibili `BV/av` 链接和中文时长表达。
- 精选视频新增兜底搜索入口：当搜索服务没有返回稳定的 Bilibili/YouTube 直链时，系统不再空白失败，而是给出平台搜索入口卡片，引导用户只选 1-2 个短讲解后回到导学闭环。
- 精选视频已纳入学习资产沉淀：从对话生成后可以保存到 Notebook，并在笔记本里继续以视频学习卡片预览。
- 精选视频观看行为已回流画像证据：用户点开公开视频时，前端会记录 `viewed external_video` 证据，并在卡片上提示“已记入画像依据”，同时刷新画像与证据缓存；后续画像可据此判断学习者是否偏好视频、短时补基或图解配套。
- 精选视频嵌入预览也可显式回写画像：当搜索结果提供 `embed_url` 时，用户可以直接在页面预览，并点击“我看了这个，记入画像”，避免只有外部跳转才产生学习证据。
- 画像服务已经把 `external_video` 识别为“精选公开视频”偏好：用户看过公开视频后，下一步补基建议会优先生成精选公开视频，而不是把视频偏好退回成普通图解。
- Notebook 保存证据已能识别资源类型：保存图解、练习、动画、研究报告或精选视频时，画像证据不再只记成普通聊天，而是带上 `visual`、`quiz`、`video`、`research`、`external_video` 等类型。
- Notebook/题目本继续去开发者化：题目快录里的记录编号、题目编号、难度和保存状态都改为学习者能理解的表达；生成摘要后保留预览，避免用户刚看到结果就被跳走。
- 图解和数学动画结果新增显式“有帮助，记入画像”按钮：用户可以主动告诉系统这份资源有效，画像会把 `visual` 或 `video` 使用偏好纳入后续推荐。
- 学习画像概览的“为什么这样安排”新增闭环提示：不仅解释当前推荐依据，也告诉用户做完后会回写掌握度和下一步建议，避免画像像静态标签。
- 导学资源生成也写入证据账本：系统会记录“生成过哪些资源”，但画像只把用户真实看过、保存过、答过或完成过的资源当作偏好信号，避免系统生成行为反向污染学习画像。
- 学习画像“依据”页开始把后台证据翻译成用户能读懂的标签，例如“看过 / 公开视频”“保存 / 图解”“答题 / 练习”，保持透明但不暴露调试日志。
- 学习画像“依据”页新增证据结论卡：先总结最近证据说明了什么、累计参考了多少记录、当前筛选里有哪些行为和资源类型，再展示明细列表，减少用户直接面对流水账的压力。
- 学习画像概览继续弱化仪表盘感：推进方式不再默认展示可信度/正确率百分比，改为“系统比较有把握 / 仍在继续校准 / 先轻量观察”等用户能直接理解的判断。
- 学习画像“推进方式”继续压缩成学习协议：用户先看到系统会按哪种方式带自己学，再看起步、推进、反馈三步策略，减少画像页的报表感。
- 导学资源推荐已识别 `external_video` 偏好：如果画像显示学习者会使用公开视频，后续学习处方和资源建议会主动给出精选公开视频，而不是只生成系统内部短视频。

## 第一阶段：统一学习画像中心

目标：把分散在导学、Memory、Notebook、题目本和会话历史里的用户信息收束成一个可展示、可更新、可解释的学习画像。

详细设计见 [学习画像设计调研与实现方案](./learner-profile-design.md)。画像是后续导学、资源推荐、多智能体调度和学习效果评估的优先中枢，开发时应先保证证据模型、可解释性和用户校准能力。

建议实现：

- 新增或强化“学习画像”页面/面板。
- 展示学习目标、当前水平、薄弱知识点、资源偏好、最近表现、系统判断依据。
- 标注画像来源，例如：来自对话、来自前测、来自练习、来自反思、来自 Notebook。
- 导学任务完成后自动更新画像摘要。
- Chat 中可引用画像作为个性化上下文。

验收标准：

- 用户能一眼看懂系统认为自己“会什么、卡在哪、适合怎么学”。
- 画像不是静态表单，而是随着对话、练习和导学证据持续更新。
- 前端能展示画像变更原因。

## 第二阶段：导学闭环重构

目标：把导学从“功能集合页”变成懒人式、递进式学习流程。

建议流程：

```text
目标确认
  -> 前测诊断
  -> 当前任务
  -> 资源生成
  -> 交互练习
  -> 提交答案/反思
  -> 系统反馈
  -> 推荐下一步
```

建议实现：

- 导学首页只展示当前任务和下一步按钮。
- 资源生成结果放在任务子页中分页展示：图解、短视频、练习、文字讲解。
- 移除不稳定或不必要的资料推荐入口，必要时改为“系统自动引用知识库”。
- 练习必须可提交，有对错反馈和解析。
- 提交页收集完成度、困难点、反思文本，写回画像和评估。

验收标准：

- 用户不需要理解系统工具，只需要跟着流程学。
- 每个任务有明确状态：待完成、进行中、已完成、需补基。
- 完成任务后系统能给出下一步建议。

## 第三阶段：学习效果评估

目标：补齐赛题第五项，把系统从“能生成资源”提升为“能评估学习效果并调整学习方案”。

建议实现：

- 新增“学习报告”页或导学内报告页。
- 汇总最近学习行为、练习正确率、任务完成情况、反思质量、Notebook 沉淀数量。
- 生成掌握度评分、薄弱点排序、风险提示和下一步建议。
- 支持按课程、知识点、时间范围查看。
- 评估结果反向影响导学路径和资源推荐策略。

验收标准：

- 报告里有清晰指标，而不是泛泛而谈。
- 能说明为什么推荐下一步任务。
- 用户完成练习或反思后，报告会变化。

## 第四阶段：多智能体协作可视化

目标：让评委和用户直观看见多智能体如何协作，而不是只在代码层存在。

建议实现：

- 在 Chat/Guide 的资源生成过程中展示轻量协作轨迹。
- 用面向用户的名称展示智能体角色：
  - 画像智能体：判断当前基础和卡点。
  - 路径规划智能体：拆解学习步骤。
  - 知识检索智能体：从知识库补充依据。
  - 图解智能体：生成 SVG、Mermaid 或图表。
  - 动画智能体：生成 Manim 短视频讲解。
  - 出题智能体：生成选择、判断、填空、简答等练习。
  - 评估智能体：批改、反馈并调整下一步。
- 默认只显示摘要，需要时再查看详细事件。

验收标准：

- 用户能理解“为什么系统给出了这些资源”。
- 多智能体轨迹不干扰学习主流程。
- 演示时能清楚体现赛题要求的“协同资源生成”。

## 第五阶段：一门完整课程 Demo

目标：为比赛演示准备稳定、完整、可复现的课程样例。

建议课程方向：

- 机器学习基础：适合展示路径规划、图解、公式、题目和案例。
- 高等数学：适合展示公式、Manim 动画、图解和分层练习。
- C++/ROS 基础：适合结合代码知识库、RAG、练习和项目式导学。

建议准备：

- 课程知识库。
- 课程章节图谱。
- 典型学习画像。
- 一条完整导学路径。
- 3 到 5 个稳定演示任务。
- 每个任务配套文字讲解、图解、练习、必要时短视频。
- 最终学习报告。

验收标准：

- 可以在 7 分钟演示视频内跑通核心流程。
- 多模态资源提前验证稳定。
- 出现模型或渲染波动时，有可展示的历史产物和 Notebook 记录兜底。

## 近期自动验收加固

- `sparkweave competition-preflight --with-build` 已成为正式提交前的推荐命令：先跑就绪检查，再跑前端生产构建，最后导出比赛提交包，减少“检查通过但前端构建失败”的赛前风险。
- GitHub Actions 已直接运行 `sparkweave competition-preflight`：每次推送都会验证“赛前检查 -> 离线材料 -> 提交包”的正式交付入口，避免只测底层脚本而漏掉 CLI 封装层。
- CI 会上传 `competition-readiness` artifact，包含结构化就绪报告和 `competition_package` 提交包快照，方便赛前从 GitHub Actions 直接下载核对。
- 新增 `scripts/render_competition_summary.py` 和 CLI `--summary` 参数：把 JSON 就绪报告压缩成一页 Markdown 摘要，CI 会直接写入 Actions Summary，让“这次提交能不能演示”不用再翻长日志。
- `competition-package` 和 `competition-preflight` 支持 `--archive`：正式归档时可同步生成 `sparkweave_competition_package.zip`，CI artifact 也会保留这份可上传提交包。
- 赛前 readiness 已把 zip 提交包纳入检查项：不仅验证目录导出，也验证可上传压缩包确实生成且非空。
- 比赛提交包新增 `index.html`：评委或队友解压后可先打开一页式材料导航，直接进入演示页、评分点证据、7 分钟讲稿、架构图、截图和运行说明。
- 提交包 HTML 本地链接纳入 readiness：`index.html` 和演示页里的截图、架构图、材料入口如果断链，CI 会直接失败。
- `scripts/check_competition_readiness.py` 已把设置页的用户化错误提示纳入就绪检查：HMAC、apikey、upstream timeout 等底层服务异常必须在前端压缩成“密钥或鉴权信息不正确”“服务响应超时”等可处理结论，并由 Playwright 用例防止原始错误重新暴露给普通用户。
- 资料库索引进度也纳入自动验收：前端默认必须提供“关键进展”里程碑视图，完整日志只能作为排查入口存在，避免普通用户被底层 SSE/WebSocket 事件流淹没。
- 对话协作轨迹纳入自动验收：最终回答出现后，前端不能再把 `stage_start`、`Thinking...`、`writing` 等调试事件展示给用户，只保留可读的智能体协作摘要、关键观察和最终回答。

## 维护约定

- 每次新增功能时，优先判断它服务于五条赛题要求中的哪一条。
- 如果一个功能不能进入学习闭环，应谨慎加入主流程。
- 前端默认面向学习者，而不是调试者。
- 调试信息、原始 JSON、运行日志默认隐藏，只在高级入口展示。
- 涉及导学、画像、评估、资源生成的改动，应同步更新本文档或相关专题文档。
