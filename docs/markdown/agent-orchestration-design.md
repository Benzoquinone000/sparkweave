# 多智能体编排设计

SparkWeave 的多智能体不是把很多按钮摆在页面上，而是让学生用一句自然语言提出学习任务后，由系统在后台自动调度合适的角色接力完成。学生不需要记住这些角色名称，只需要知道自己下一步要学什么、看什么资料、做什么练习；背后负责分工的是对话协调器、能力路由、Agentic RAG 和工具层。课程助教则把这种能力延长到一门课的持续陪伴里，并可通过 QQ 等消息通道进入学生常用入口。

## 一次学习请求如何流转

以“我想补一下梯度下降的直观理解，先看图解再做练习”为例，系统会按下面的顺序处理：

| 步骤 | 系统动作 | 学生能看到什么 |
| --- | --- | --- |
| 1. 读取上下文 | 合并当前问题、会话历史、课程资料、学习画像和记录 | 回答会贴合当前课程和薄弱点 |
| 2. 自动判断任务类型 | 判断这次更像解释、解题、出题、图解、研究、视频素材还是资料检索 | 不需要学生手动选择智能体 |
| 3. 分配给专业能力 | 例如图解交给可视化能力，练习交给出题能力，资料依据交给 Agentic RAG | 前端展示“正在查资料、生成图解、生成练习”等过程 |
| 4. 调用工具 | 按需使用资料库、联网搜索、讯飞公式识别、图片理解、工作流等工具 | 结果里保留来源、工具记录或资源卡片 |
| 5. 汇总为学习结果 | 把图解、练习、答案、证据和下一步建议整理成可读结果 | 学生获得一份能继续学习的材料 |
| 6. 写回记录 | 学习行为、练习反馈和资源使用进入记录和画像 | 下一次推荐会受这次学习影响 |

这个流程服务赛题里的“多智能体协同资源生成”和“个性化学习路径规划”。系统重点不是展示有多少 Agent，而是把一次学习任务拆给合适的角色。

## 主要角色

| 角色 | 负责什么 | 典型页面 |
| --- | --- | --- |
| 对话协调器 | 接收学生问题，判断是否需要资料、图解、练习、解题或研究 | [问问题页](../../web/screenshots-chat.png) |
| 资料检索 | 从课程资料库和外部资料中找依据 | [资料页](../../web/screenshots-knowledge.png) |
| 深度求解 | 处理推导、证明、计算和步骤讲解 | 问问题页 |
| 练习生成 | 生成选择、判断、填空、简答等练习，并给出解析 | 问问题页、学习页 |
| 知识图解 | 把抽象概念转成流程图、关系图或结构图 | 问问题页 |
| 数学动画 | 为公式、几何含义或变化过程生成动画素材 | 问问题页 |
| 学习研究 | 拆解复杂主题，整理学习报告或学习路径 | 问问题页 |
| 学习画像与评估服务 | 读取目标、偏好、薄弱点和练习反馈，影响下一步推荐 | [记录与画像页](../../web/screenshots-memory.png) |
| 课程助教 | 围绕一门课程长期保存资料、提醒和问答上下文，可通过 QQ 等通道接入学生常用入口 | [课程助教页](../../web/screenshots-agents.png) |

## 当前内置能力

| 能力 | 面向学生的说法 | 产出 |
| --- | --- | --- |
| `chat` | 自动导学 | 普通问答、资料问答、工具调用、智能体分配 |
| `deep_solve` | 深度求解 | 分步解题、证明、推导、验证 |
| `deep_question` | 练习生成 | 题目、答案、解析、可反馈练习 |
| `deep_research` | 学习研究 | 主题拆解、资料整理、学习报告 |
| `visualize` | 知识图解 | Mermaid、SVG、图表或结构化图解 |
| `math_animator` | 数学动画 | 动画工程文件、动画素材、讲解摘要 |

这些能力在后端由 `sparkweave/app/facade.py` 定义，在 `sparkweave/runtime/runner.py` 中执行到对应的图流程。

## 工具层

专业能力在需要时会调用工具。工具不直接决定学习流程，只负责完成一个明确动作。

| 工具 | 用途 | 赛题价值 |
| --- | --- | --- |
| `rag` | 检索课程资料库 | 让回答带有资料依据 |
| `web_search` | 查找公开网页资料 | 补充课程外背景 |
| `external_video_search` | 推荐公开学习视频 | 支持多模态资源推荐 |
| `external_image_search` | 推荐公开图片或示意图 | 支持图解和视觉材料 |
| `iflytek_workflow` | 调用讯飞星辰工作流 | 接入讯飞智能体工作流能力 |
| `iflytek_formula_ocr` | 识别图片中的公式 | 支持数学题图和手写公式辅导 |
| `iflytek_image_understanding` | 理解板书、截图、示意图 | 支持图片输入后的学习辅导 |
| `code_execution` | 执行 Python 计算 | 辅助验证计算和代码题 |
| `paper_search` | 查找学术论文 | 支持研究型学习任务 |
| `reason` | 专门推理调用 | 用于复杂推导或解释 |

工具注册和别名集中在 `sparkweave/tools/builtin.py` 与 `sparkweave/tools/registry.py`。

## 自动调度怎么做

自动调度不是把关键词硬匹配到菜单。一次请求进入后，`context_enrichment.py` 会先把用户问题、会话历史、课程资料、学习画像、笔记上下文和运行配置整理成 `UnifiedContext`。随后 `LearningCapabilityRouter.decide()` 先用规则做低成本判断：是否强制指定能力，是否关闭自动委派，是否已经委派过一次，是否明确说了“不联网 / 不用工具 / 不开画布”，以及本轮更像解释、解题、出题、图解、动画、研究还是外部资源搜索。

规则判断置信度不够时，路由器还可以调用 LLM intent coordinator。它不会直接回答学生，而是返回严格 JSON：目标 capability、可用工具、是否使用画像提示、置信度、原因和可选的重写任务。这样做的好处是调度逻辑既有确定规则，也能处理“我想先补一下这个，再找点材料练一练”这类边界不清的学习请求。

| 调度线索 | 可能进入的能力 |
| --- | --- |
| “解释、为什么、举例、比较” | 普通对话 + Agentic RAG 资料依据 |
| “证明、推导、计算、步骤” | `deep_solve` 深度求解 |
| “出题、练习、测验、检查掌握” | `deep_question` 练习生成 |
| “画图、结构图、流程图、关系图” | `visualize` 知识图解 |
| “动画、视频讲解素材、Manim” | `math_animator` 数学动画 |
| “继续学习、下一步、薄弱点” | 学习画像 + 导学服务 |

调度结果会继续进入 `delegated_context()`：系统把本轮任务、工具列表、重写后的 prompt 和路由原因写进 metadata，再交给 `deep_solve`、`deep_question`、`deep_research`、`visualize`、`math_animator` 或默认 `chat` 图流程。前端的 `AgentCollaborationPanel` 读取这些事件，展示“画像 -> 协调器 -> 专业智能体 -> 工具”的协作过程。

| 技术点 | 代码里的处理 |
| --- | --- |
| 显式约束优先 | 用户说“不要联网”“不用工具”“不要画布”时，路由器会裁剪对应工具，而不是强行展示能力 |
| 直接工具与专业能力分开 | 找公开视频、找参考图片走 `external_video_search` / `external_image_search`，生成图解或动画才交给 `visualize` / `math_animator` |
| 画像只在有用时介入 | “继续学习”“下一步做什么”这类模糊请求会读取 `learner_profile_context`，证据弱时不强行个性化 |
| 协作路线可解释 | `collaboration_route()` 会给前端返回画像、协调器和专业智能体的步骤说明 |
| RAG 内部继续 agentic | 资料相关任务进入 `rag` 工具后，复杂问题再由 Agentic RAG 拆分、合并、质检和回退 |

## 课程助教与 QQ 通道

课程助教负责更长周期的学习陪伴：它可以保存课程文件、助教人格、历史对话、技能、提醒任务和工作区笔记。Web 端用于创建和管理助教，消息通道层则把助教接到外部入口。当前通道配置包含 `qq`，代码里有 `QQChannel`，支持 QQ 私聊、群聊消息和定时提醒目标；如果没有配置真实 QQ 凭据，通道会退回到内存通道，保证系统管理能力仍可演示。

更具体一点，`QQConfig` 定义 `enabled`、`app_id`、`secret`、`allow_from` 和 `msg_format`；`QQChannel` 启动时加载 `botpy`，监听私聊、直接消息和群 at 消息，收到内容后统一整理成 `session_key=qq:<chat_id>` 的助教会话。发送提醒时，定时任务可以指定 channel 和目标，QQ 通道再按群聊或私聊调用对应发送接口。这样网页里的长期助教、提醒任务和 QQ 入口没有分成两套系统。

| 能力 | 代码位置 |
| --- | --- |
| 课程助教管理 API | `sparkweave/api/routers/sparkbot.py` |
| 助教运行管理 | `sparkweave/services/sparkbot.py` |
| 通道配置模型 | `sparkweave/services/sparkbot_support/config_models.py` |
| QQ 通道实现 | `sparkweave/services/sparkbot_support/channels.py` |
| 前端课程助教页面 | `web/src/pages/agents/` |
| 课程助教依赖 | `requirements/sparkbot.txt` |

答辩时可以这样讲：网页是教师或团队维护助教的地方，QQ 是学生日常触达助教的入口。这样“智能辅导”不只停留在演示页面里，也能覆盖课后提醒、群内答疑和阶段复盘。

## 路由原则

系统不会为了展示能力而强行走复杂流程。当前路由遵循几条朴素规则：

- 学生只是问一个概念，优先直接解释。
- 学生要求“证明、推导、计算”，进入深度求解。
- 学生要求“出题、练习、测验”，进入练习生成。
- 学生要求“画图、流程图、知识结构”，进入图解。
- 学生要求“动画、视频讲解素材、Manim”，进入数学动画。
- 学生要求“找公开视频、公开课、参考图片”，优先调用搜索工具。
- 学生说“继续学习、下一步做什么”，系统会读取学习画像和当前课程状态。
- 学生明确说“不用工具、直接回答、不联网”，本轮会尊重这个限制。

路由策略主要在 `sparkweave/runtime/capability_router.py` 和 `sparkweave/graphs/chat.py` 中实现。

## 前端如何呈现协作过程

前端不会把处理过程拆成复杂面板。它主要呈现三类信息：

| 信息 | 作用 |
| --- | --- |
| 进度状态 | 告诉学生系统正在查资料、生成练习、整理图解或写回答 |
| 证据来源 | 让资料问答能追溯到课程文件或外部来源 |
| 结果卡片 | 展示题目、图解、视频推荐、图片推荐、语音或工具结果 |

对应前端位置包括 `web/src/hooks/useChatRuntime.ts`、`web/src/components/chat/MessageBubble.tsx`、`web/src/components/results/` 和 `web/src/components/chat/AgentCollaborationPanel.tsx`。

## 与赛题的关系

| 赛题点 | 多智能体编排中的对应 |
| --- | --- |
| 对话式学习画像 | 画像内容进入请求上下文，影响继续学习和资源推荐 |
| 多智能体资源生成 | 协调器按上下文自动把任务交给解题、出题、图解、动画、研究等能力 |
| 个性化路径规划 | 学习页和导学服务根据画像与课程状态生成当前任务 |
| 智能辅导 | 问题进入对话协调器后，可同时使用资料、图解、公式识别、图片理解、练习和课程助教通道 |
| 学习效果评估 | 练习和反馈写入记录，影响后续路径和画像 |
| 讯飞工具使用 | 讯飞工作流、公式识别、图片理解、语音和模型配置作为工具链接入 |

## 边界说明

- 多智能体编排依赖当前仓库内置能力，不把第三方插件能力作为核心演示前提。
- 自动路由会受本轮可用工具、用户明确限制和后端安全策略约束。
- QQ 等外部通道需要真实平台凭据和相应依赖；没有配置时，系统仍可在 Web 端展示课程助教管理、提醒和工作区能力。
- 数学动画和多模态素材生成可能受本地依赖、模型配置和运行时间影响，演示时可准备已生成的图解或动画结果作为补充。
- 资料问答的质量取决于课程资料和向量库状态；没有资料库时，系统会按普通问答呈现，不标注为资料引用。
