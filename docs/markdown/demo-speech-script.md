# SparkWeave 项目展示台词稿

这版按“功能亮点 + 技术创新点”来讲，适合正式录制演示视频。主线不要散：先证明资料问答有证据，再展示题目生成、图片/图解生成、动画生成，最后用课程助教收住长期学习场景。

建议录制顺序：

1. 项目首页：一句话定位。
2. 资料库 / RAG：证明课程资料能检索、回答有来源。
3. 题目生成：证明不是只会答题，还能生成练习和解析。
4. 图片 / 图解生成：证明能把知识转成可视化资源。
5. 动画生成：证明能生成动态讲解资源。
6. 课程助教：证明能力可以长期陪伴，并可接 QQ 等外部入口。
7. 学习画像 / 记录：证明所有结果能回到学习证据和下一步建议。

录制前先准备好：

- Docker 前端：`http://127.0.0.1:3782`
- 后端：`http://127.0.0.1:8001`
- 课程资料库：优先展示“深度学习”，备用展示“智能机器人系统”
- 问问题页：确认模型、资料库和工具状态正常
- 设置页：不要展示完整 API Key

## 0:00-0:30 开场：项目定位

页面动作：打开项目首页。

台词：

大家好，我们展示的作品是 SparkWeave，中文名是星火织学。它面向软件杯 A3 赛题，做的是一个高校课程场景下的个性化学习多智能体系统。

这套系统不是单纯的聊天机器人。我们的设计目标是让学生围绕一门真实课程完成学习闭环：课程资料进入知识库，RAG 给出有来源的解释，多智能体生成练习、图解和动画，课程助教长期跟进，最后学习记录再回写到画像和下一步建议里。

接下来我不按代码模块讲，而是按评委最容易核验的功能来演示：RAG、题目生成、图片和图解生成、动画生成、课程助教，以及背后的技术创新点。

页面动作：点击“进入主页面”或直接进入“资料”页。

## 0:30-1:40 Agentic RAG：资料问答不是一句空回答

页面动作：进入“资料”页，选择“深度学习”资料库，展示文档、入库状态、检索测试或来源区域。

推荐提问：

```text
请基于深度学习课程资料，解释卷积层和池化层分别解决什么问题，并说明它们为什么适合图像任务。回答时请保留资料依据。
```

台词：

先看资料库。这里的“深度学习”不是临时写的一组问答，而是由课程课件同步进来的知识库。系统还接入了第二门“智能机器人系统”课程，用来证明课程模板和资料库可以迁移到不同专业课。

我现在提一个和 CNN 相关的问题。普通 RAG 通常是一次检索加一次回答，但 SparkWeave 这里做的是 Agentic RAG。

它的技术流程是：先根据问题生成 query plan，把一个复合问题拆成多个检索方向；然后并发检索课程资料；再合并来源，计算证据质量，比如来源数量、覆盖度、相关性和上下文长度；如果证据不足，会尝试修复检索，仍然不足时会保守回答，而不是硬编。

所以这里要看两个点：第一，回答是否围绕课程资料；第二，旁边或下方是否能展开来源。这个证据链是我们和普通聊天工具最大的区别之一。

如果模型等待：

现在后台正在做检索计划、资料召回和证据合并。这里等待的不是单纯大模型输出，而是 RAG agent 在组织资料证据。

结果出来后：

这里可以看到，回答不是孤立文本，而是带着来源和检索过程。对学生来说，它能知道答案来自哪份课件；对评委来说，这个结果可以现场核验。

技术创新点补一句：

这部分的创新点是把 RAG 做成可观测的智能体流程，而不是黑盒检索。系统把 query plan、质量报告、修复记录和 sources 都返回给前端，方便解释和追责。

## 1:40-2:40 题目生成：从资料到练习，而不是只会讲解

页面动作：进入“问问题”页，能力选择“题目生成”；或者进入“练习 / 题目生成”页面。

推荐输入：

```text
请基于深度学习资料库，围绕 CNN 的卷积、池化、特征图生成 5 道中等难度练习题。题型混合，包含答案、解析和对应知识点。
```

台词：

接下来展示题目生成。学习系统不能只回答问题，还要能让学生练起来。

这里我让系统围绕 CNN 生成 5 道练习题。题目生成不是简单让模型随便编题，而是和课程资料、当前学习目标、难度、题型偏好绑定在一起。结果里会包含题干、答案、解析和知识点标签。

背后对应的是 `deep_question` 练习生成智能体。它会经历需求理解、题目生成、质量检查和结果整理。生成后的题目可以进入题目本，也能作为学习效果评估的证据。

如果要讲得更技术一点：

题目生成的价值在于把“资源生成”和“效果评估”连起来。一次练习不是生成完就结束，学生的作答、错因、知识点和难度会继续进入学习证据，影响后面的薄弱点判断和下一步推荐。

结果出来后：

这里可以看到题目不只是列表，还带答案和解析。录制时可以点开一道题，强调它能从课程资料延伸成可训练、可反馈的学习任务。

技术创新点补一句：

这部分创新点是“生成即评估入口”。题目不是一次性文本产物，而是能进入题目本、学习记录和画像更新的结构化资源。

## 2:40-3:40 图片 / 图解生成：把知识变成可读的视觉资源

页面动作：进入“问问题”页，能力选择“知识可视化”；或者在自动导学里输入图解请求。

推荐输入：

```text
请生成一张面向初学者的 CNN 结构图，展示输入图像、卷积核、特征图、池化层、全连接层和分类结果之间的关系，并用一句话说明读图顺序。
```

台词：

接下来是图片和图解生成。这里我需要先说清楚：SparkWeave 默认展示的是面向学习的图解生成和精选图片资源，不把它包装成通用文生图工具。我们的目标不是生成一张漂亮海报，而是生成学生能看懂、能复习、能保存的学习资源。

我现在让系统生成 CNN 的结构图。后台会先分析可视化需求，判断适合 Mermaid、SVG、流程图还是结构图，再生成可渲染代码和说明。

这里有一个细节：可视化智能体不是只生成代码就结束，它会做检查。如果 Mermaid 或 SVG 渲染失败，系统会把错误反馈给智能体进行修改。也就是说，图解生成有“报错反馈和自修复”机制，避免把坏图交给学生。

如果图生成完成：

现在可以看到图解把 CNN 的输入、卷积、池化和分类流程串起来了。这个结果可以保存到笔记，也能成为后续学习记录的一部分。

如果想展示“精选图片”：

如果我需要公开图片素材，也可以让系统找“CNN 结构示意图参考图片”。这会走 `external_image_search`，返回公开图片资源和来源，用来辅助学生理解，而不是把来源不明的图片混进课程资料里。

技术创新点补一句：

这部分创新点是“可渲染资源生成”。系统生成的是带源码、类型、说明和校验过程的学习图解，不只是大模型随口描述一张图。

## 3:40-4:55 动画生成：把抽象过程做成动态讲解

页面动作：进入“问问题”页，能力选择“数学动画”；或者 URL 使用 `/chat?capability=math_animator`。

推荐输入：

```text
请生成一段 30 秒左右的教学动画，用二维曲线解释梯度下降如何从初始点一步步靠近损失函数最低点。要求包含坐标轴、当前位置、小步迭代箭头和简短旁白。
```

台词：

接下来展示动画生成。很多深度学习概念，比如梯度下降、反向传播、注意力机制，只靠文字解释会比较抽象，所以系统提供了动画生成能力。

我这里让它生成一段梯度下降动画。后台对应的是 `math_animator` 能力，它会把自然语言需求转成动画脚本和视频结果。配置里可以控制输出模式、质量、是否生成旁白、风格提示和最大重试次数。

这不是简单调用一个视频模板。它的流程是：先分析动画需求，再生成 Manim 风格的代码，再渲染视频；如果代码运行失败，会进入 retry manager，根据报错修改代码；渲染完成后还可以做视觉检查，确认画面是不是空白、元素有没有明显错位。

如果生成较慢：

动画生成会比普通问答慢，因为它涉及代码生成、渲染和错误修复。录制时这里可以不用等完整视频结束，可以展示任务过程和已生成的脚本或结果预览。

结果出来后：

这里可以看到，系统把一个抽象数学过程变成了动态学习材料。对于学生来说，这比一段文字更容易理解；对于课程建设来说，它能快速生成配套讲解资源。

技术创新点补一句：

这部分创新点是“代码驱动的多模态资源生成”。动画不是模型直接吐视频，而是由智能体生成可执行代码，通过渲染、报错反馈和重试得到可复用资源。

## 4:55-5:55 课程助教：把一次问答延长成长期陪伴

页面动作：进入“课程助教”页，展示深度学习课程助教、工作区、提醒任务、通道配置。

台词：

前面展示的是一次学习任务里的能力。接下来是课程助教，它负责把这些能力延长到一门课的长期学习中。

课程助教可以保存课程资料、助教人格、历史对话、工作区文件和提醒任务。比如老师可以创建一个“深度学习课程助教”，让它围绕这门课长期答疑、提醒复习、整理资料。

这里还有一个面向真实使用场景的设计：助教可以接外部消息入口，代码里已经有 QQ 通道模型和处理链路。它支持私聊、群聊消息和定时提醒目标。如果现场没有配置真实 QQ 凭据，我们只展示配置入口和 Web 试问，不会把未配置状态说成真实上线。

技术上，课程助教不是普通 chat session。每个助教有自己的配置、workspace、技能、cron 任务和历史记录。这样它可以围绕一门课程持续工作，而不是每次都从零开始。

技术创新点补一句：

这部分创新点是“学习智能体的长期化”。系统不只在网页里回答一次，而是把课程资料、提醒、通道和工作区组织成一个可持续运行的课程助教。

## 5:55-6:35 记录与画像：让生成结果回到学习闭环

页面动作：进入“记录 / 画像”页，展示学习证据、薄弱点、下一步建议。

台词：

最后看记录和画像。前面生成的回答、题目、图解、动画和助教互动，如果只是停留在页面上，就还是一次性工具。SparkWeave 的设计是让这些学习行为回到证据账本里。

系统会记录学生问了什么、用了哪些资料、保存了哪些资源、做了哪些练习、反馈了什么结果。画像服务再从这些证据里聚合出当前重点、薄弱点、学习偏好和下一步建议。

这里的个性化不是一开始让学生填标签，而是跟着学习过程不断更新。比如学生在 CNN 题目上反复出错，后续系统就可以推荐补基练习、图解或动画，而不是继续推同样难度的内容。

技术创新点补一句：

这部分创新点是“证据化画像”。画像不是模型随便总结，而是由学习证据、题目结果、资料使用和学生校准共同驱动，并且运行时只注入精简画像提示，避免上下文被画像压满。

## 6:35-7:00 收尾：把创新点串起来

页面动作：回到学习页或项目首页。

台词：

总结一下，SparkWeave 的核心不是堆很多 AI 功能，而是把它们组织成一条可核验的学习链路。

第一，Agentic RAG 让资料问答有计划、有来源、有质量判断。第二，题目生成把课程内容变成可练习、可反馈的结构化资源。第三，图解和动画生成把抽象知识变成多模态学习材料，并带有渲染检查和错误修复。第四，课程助教把这些能力延长到长期学习和 QQ 等外部入口。第五，所有学习行为再回到记录和画像，影响下一步路径。

这就是我们对 A3 赛题的实现思路：用大模型和多智能体，不是替学生多写几段答案，而是让课程资料、资源生成、智能辅导和学习评估形成一个可以运行、可以核验、可以持续优化的系统。

## 评委追问时的回答

如果问：你们的 RAG 和普通 RAG 有什么区别？

可以答：

普通 RAG 通常是一次检索加回答。我们这里是 Agentic RAG，会先生成 query plan，把复杂问题拆成多个子查询，并发检索后合并来源，再用来源数、覆盖度、相关性和上下文长度形成质量报告。证据不足时会修复检索或保守回答，前端也能看到来源和检索过程。

如果问：题目生成怎么体现个性化？

可以答：

题目生成会结合课程资料、当前知识点、难度和题型配置。更重要的是，题目不是一次性文本，学生作答后的正确率、错因、知识点和难度会进入学习证据，影响画像里的薄弱点和下一步推荐。

如果问：图片生成到底是什么？

可以答：

我们默认展示的是学习图解生成和精选图片资源。知识可视化智能体会生成 Mermaid、SVG 或结构图，并进行渲染校验和错误修复；精选图片工具会找公开图片并保留来源。如果现场配置了讯飞工作流，也可以把图像生成工作流接入，但默认不把它说成通用文生图。

如果问：动画生成是不是模板？

可以答：

不是固定模板。动画生成会把自然语言需求转成 Manim 风格代码，渲染视频。如果代码报错，会根据错误反馈重试；渲染后还可以做视觉检查。它更像一个“代码生成 + 执行 + 修复”的多智能体流程。

如果问：课程助教和普通聊天有什么区别？

可以答：

普通聊天主要是一条会话。课程助教有独立配置、课程资料、workspace、技能、历史记录、cron 提醒和通道配置，可以围绕一门课长期运行。代码里有 QQ 通道模型和消息处理链路，没有真实凭据时会退回 Web 或内存通道演示。

如果问：技术创新点是什么？

可以答：

我们不是单点做问答，而是把五个技术点串起来：Agentic RAG 证据链、自动调度多智能体、代码驱动的图解和动画生成、证据化学习画像、可长期运行的课程助教。它们共同服务一条学习闭环，而不是散装功能展示。

## 录屏节奏建议

- RAG 一定要展示来源，不然创新点会弱。
- 题目生成至少点开一道题，展示答案和解析。
- 图解生成如果现场不稳定，就展示生成过程和已有结果，不要长时间等待。
- 动画生成可能耗时，正式录屏可以先生成一次，现场展示过程和结果。
- 课程助教展示配置、提醒和通道即可，不要强行配置真实 QQ。
- 设置页只展示服务类型和连接状态，不展示完整密钥。
