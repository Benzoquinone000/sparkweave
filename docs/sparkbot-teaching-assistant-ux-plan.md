# SparkWeave AI 助教中心优化计划书

> 版本：v0.1  
> 日期：2026-05-14  
> 范围：助教功能产品体验、前端信息架构、SparkBot 能力封装、比赛演示闭环  
> 目标页面：`/agents`，现阶段对应文件主要为 `web/src/pages/AgentsPage.tsx`

## 1. 项目背景

本项目面向科大讯飞赛题，核心任务是构建高等教育个性化学习资源体系和智能学习智能体系统。赛题要求覆盖：

- 对话式学习画像自主构建。
- 多智能体协同的资源生成。
- 个性化学习路径规划和资源推送。
- 智能辅导，作为可选加分项。
- 学习效果评估，作为可选加分项。
- AI 辅助工具需选用科大讯飞相关工具。
- 自行构造至少一门完整高校专业课程。

SparkWeave 已经具备较强的 agent-native 技术基础：统一 Chat runtime、RAG、学习画像、学习效果评估、能力图、SparkBot 长期智能体、工作区文件、技能、渠道、定时任务和多智能体协作能力。当前问题不是能力不足，而是助教入口仍偏“后台管理台”，学生视角下的主线不够清晰。

本计划书的核心结论是：将 `/agents` 从“SparkBot 管理页面”升级为“AI 助教中心”，默认面向学生的学习任务，而不是面向开发者的 Bot 配置。

## 2. 建设目标

### 2.1 产品目标

AI 助教中心要让学生进入页面后立刻知道三件事：

1. 我现在应该学什么。
2. 为什么系统建议我这样学。
3. 学完后系统会如何记录、评估并调整下一步。

助教不是普通聊天框，也不是后台 Bot 控制台。它应该是一个持续在线、能读取学习画像、能调用课程资料、能生成多模态资源、能安排练习复盘的个性化学习伙伴。

### 2.2 比赛目标

助教优化应直接服务评分项：

| 评分项 | 助教中心贡献 |
| --- | --- |
| 创新价值与实用性 35% | 展示长期助教、个性化学习画像、来源可追溯答疑、多模态资源和闭环评估 |
| 功能实现及技术要求 45% | 展示 SparkBot、多智能体能力图、RAG、讯飞星火、语音识别、语音合成、OCR 等能力 |
| 配套文档 10% | 产出本计划书、实现说明、演示脚本、AI Coding 说明 |
| 演示视频、PPT 10% | 形成 7 分钟内可跑通的助教演示主线 |

### 2.3 用户体验目标

- 默认视图回答“现在做什么”，不让学生先理解 Bot、JSON、channel、runtime。
- 保留 SparkBot 的高级能力，但把高级配置折叠到二级入口。
- 所有过程状态使用学习语言，例如“正在查看课程资料”“正在生成练习”“正在写入学习记录”。
- 每次助教回答后给出下一步动作，而不是让学生重新想 prompt。
- 每条关键建议都能说明依据：课程资料、学习画像、练习结果、最近反馈。
- 视觉风格遵循 `DESIGN-notion.md`：白色画布、细边框、低噪声、8px 以内圆角、Notion 式工作台感。

## 3. 当前现状判断

### 3.1 已有能力

当前 SparkBot 已经具备以下基础：

- 独立长期助教实例：每个 Bot 有独立配置、工作区文件、会话历史、技能、渠道和后台任务。
- 工作区上下文：`SOUL.md`、`USER.md`、`TOOLS.md`、`AGENTS.md`、`HEARTBEAT.md` 参与 prompt 构建。
- 长期记忆：可读取全局 `PROFILE.md`、`SUMMARY.md` 和 Bot 私有 memory。
- 多工具体系：RAG、web search、code execution、reason、brainstorm、paper search、workspace 文件工具、MCP 工具。
- 后台能力：`/btw` side task、`/cron` 定时任务、`/team` nano-team。
- 前端入口：`AgentsPage.tsx` 已有助教列表、创建表单、聊天、能力卡、工作区文件、渠道配置和高级 JSON 编辑。

### 3.2 主要体验问题

| 问题 | 影响 |
| --- | --- |
| 页面定位偏管理台 | 学生看到的是“管理助教/创建助教/高级配置”，不是学习任务 |
| 首屏缺少下一步学习建议 | 不能直接展示个性化推荐和画像价值 |
| 创建助教是表单，不是向导 | 用户需要理解标识、persona、模板等概念 |
| 命令能力隐藏在文本命令里 | `/new`、`/btw`、`/cron`、`/team` 对普通学生不可见 |
| 过程状态偏技术化 | 不利于展示智能体协作价值 |
| 资料和产物没有学习化组织 | 工作区文件存在，但缺少“讲解卡/练习卡/错题复盘/学习记录”视角 |
| 高级配置和学生任务混在一个页面 | 认知负担高，比赛演示也容易跑偏 |
| `NOTES.md` 存在前后端不一致 | 前端提示可创建 `NOTES.md`，后端 allowlist 当前不允许 |

### 3.3 技术边界

SparkBot 与普通 `/api/v1/ws` one-turn runtime 是两套入口。助教中心第一阶段不强行重写 runtime，而是在当前 SparkBot 能力上做用户体验封装：

- 保留 SparkBot 长期实例能力。
- 前端将 Bot 管理能力重新包装为学习助教工作台。
- 对话中通过快捷动作、结构化状态和反馈按钮引导学习闭环。
- 当前已将学习画像、学习效果和课程包能力接入 SparkBot 演示主线，作为 AI 助教中心的默认闭环。

## 4. 落地产品参考

本计划参考了已经落地的 AI 学习产品，重点提炼可借鉴的交互模式，而不是照搬视觉。

### 4.1 Khanmigo

Khanmigo 将 AI 定位为 personal tutor 和 teaching assistant。其学习侧强调引导学生思考，而不是直接给答案；教师侧提供任务型工具，例如 lesson plan、rubric generator、recommend assignments、class snapshot 等。

对 SparkWeave 的启发：

- 助教要有“苏格拉底式追问”和“直接讲解”两种模式。
- 不同学习任务应做成卡片或按钮，而不是让学生记 prompt。
- 教师/评审演示需要看到班级洞察、学习记录和推荐依据。

参考：[Khanmigo 官方页面](https://www.khanacademy.org/khan-labs)、[Khanmigo teacher tools](https://support.khanacademy.org/hc/en-us/articles/14799047733645-What-teacher-tools-are-available-on-Khanmigo)

### 4.2 NotebookLM

NotebookLM 的核心优势是资料约束和产物生成。它围绕 sources 组织问答，并提供 citations、Audio Overviews、Video Overviews、Mind maps、Reports、Flashcards、Quizzes、Slide Decks、Infographics 等学习产物。

对 SparkWeave 的启发：

- 助教回答必须尽量显示来源依据，尤其是 RAG 命中资料。
- “资料库”和“助教产物”要成为学生可见资产，而不是隐藏在工具日志里。
- 一个问题的结果可以生成多种学习格式：文字讲解、图解、练习、摘要、复习卡。

参考：[NotebookLM 创建 notebook](https://support.google.com/notebooklm/answer/16206563?hl=en)、[NotebookLM chat citations](https://support.google.com/notebooklm/answer/16179559?hl=en-GB)

### 4.3 Duolingo Max

Duolingo Max 将 AI 能力嵌入学习路径旁边，而不是做成孤立聊天入口。其 Roleplay、Video Call、Explain My Answer 都围绕“刚学过的内容”和“做错后的反馈”展开。

对 SparkWeave 的启发：

- 助教要贴近当前学习路径，而不是等待用户主动提问。
- 每次练习后应给“为什么错”和“下一次怎么做”的轻量反馈。
- 对话训练、情景模拟、视频/语音讲解适合做成路径旁的支线任务。

参考：[Duolingo Max 官方介绍](https://blog.duolingo.com/duolingo-max/)、[Explain My Answer](https://blog.duolingo.com/explain-my-answer-now-free/)

### 4.4 Coursera Coach

Coursera Coach 强调在课程学习过程中提供 on-demand assistance，支持概念解释、摘要、练习题和职业场景应用。Coursera 的企业学习页面也强调 AI 辅助课程构建和定制学习路径。

对 SparkWeave 的启发：

- 助教要嵌入课程上下文，默认知道当前课程、章节和学习目标。
- 资源推荐要解释“为什么推荐”，并能联系真实应用场景。
- 面向比赛演示时，应突出“课程内智能辅导”和“个性化路径”。

参考：[Coursera Coach 官方博客](https://blog.coursera.org/coursera-coach-providing-essential-learner-support/)、[Coursera customized learning](https://www.coursera.org/enterprise/customized-learning)

### 4.5 科大讯飞能力

比赛要求使用科大讯飞相关工具。助教中心应明确使用并展示以下能力：

- 讯飞星火大模型：作为主模型或可选模型提供对话、讲解、资源生成。
- 星火知识库或现有 RAG 接入：用于课程资料问答和来源约束。
- 讯飞语音听写：用于学生语音提问、课堂录音转写、口述笔记。
- 讯飞语音合成：用于短音频讲解、视频讲解配音。
- 讯飞 OCR/公式识别：用于拍照题目、板书、讲义截图解析。
- 讯飞 embedding 或兼容服务：用于知识库向量化和学习资源检索。

参考：[讯飞开放平台文档中心](https://www.xfyun.cn/doc/)、[星火大模型可定制 API](https://www.xfyun.cn/doc/spark/OptionalAPI.html)、[星火知识库 API](https://www.xfyun.cn/doc/spark/ChatDoc-API.html)、[语音听写流式 API](https://www.xfyun.cn/doc/asr/voicedictation/API.html)、[在线语音合成](https://www.xfyun.cn/doc/tts/online_tts/Java-SDK.html)

## 5. 产品定位

### 5.1 新定位

页面名称：AI 助教中心

一句话定位：

> 面向高校课程的长期学习助教，能根据学习画像继续学习、基于课程资料答疑、协同多个智能体生成资源，并把练习和反馈写回学习效果评估。

### 5.2 用户角色

| 角色 | 主要诉求 |
| --- | --- |
| 学生 | 今天学什么、哪里不会、怎么补、有什么资源、练完是否进步 |
| 教师 | 学生薄弱点、资源生成质量、学习过程证据、课程完成情况 |
| 评审 | 系统是否多智能体、是否个性化、是否多模态、是否使用讯飞工具、是否可运行 |
| 开发者 | Bot 配置、渠道、工具、日志、prompt、runtime 参数 |

默认体验服务学生和评审，开发者入口保留但后置。

## 6. 演示课程方案

建议比赛主课程采用：

课程名：大模型与智能学习系统  
课程标识：`ai_learning_agents_systems`  
适用对象：计算机、人工智能、教育技术相关专业本科高年级或研究生  
课程周期：8 周或 10 周  
课程目标：理解大模型、RAG、多智能体、学习画像、资源生成和学习评估闭环，并能构建一个教育智能体原型。

### 6.1 课程模块

| 周次 | 模块 | 助教可展示能力 |
| --- | --- | --- |
| 1 | 大模型基础与提示工程 | 概念讲解、错因追问、学习画像初始化 |
| 2 | RAG 与知识库构建 | 来源约束问答、引用证据链 |
| 3 | 多智能体协作机制 | 协作路线可视化、任务拆分 |
| 4 | 个性化学习画像 | 对话式画像构建、偏好校准 |
| 5 | 学习资源生成 | 图解、练习、讲义、复习卡生成 |
| 6 | 智能答疑与多模态交互 | 语音提问、OCR 题目识别、TTS 讲解 |
| 7 | 学习效果评估 | 概念掌握度、错因闭环、下一步计划 |
| 8 | 课程项目实践 | 项目助教、评审清单、答辩材料生成 |

### 6.2 课程资料包

至少准备以下资料：

- 课程大纲：学习目标、章节、知识点、作业。
- 讲义 Markdown 或 PDF：每章 1 份。
- 示例题库：选择题、简答题、代码题、案例题。
- 资源索引：视频、论文、案例、拓展阅读。
- 评分标准：项目 rubric、实验报告模板。

助教中心演示时绑定该课程包，避免空泛答疑。

## 7. 信息架构规划

### 7.1 顶层结构

将当前 `/agents` 的标签从技术视角重构为学习视角：

| 新入口 | 当前可复用区域 | 说明 |
| --- | --- | --- |
| 我的助教 | 助教列表、最近活动、聊天 | 默认首页，展示当前助教和今日建议 |
| 助教对话 | `SparkBotChat` | 课程上下文答疑、快捷动作、来源和反馈 |
| 助教能力 | agent capabilities | 以任务卡展示解题、出题、图解、调研、复盘 |
| 资料与产物 | workspace files、history | 学习资料、生成产物、笔记、错题、复习卡 |
| 学习记录 | learner profile、learning effect | 展示画像证据、掌握度、下一步行动 |
| 渠道与高级 | channels、tools、runtime JSON | 开发者和演示准备入口，默认折叠 |

### 7.2 首屏布局

首屏应从“管理助教”改为“当前学习工作台”：

1. 顶部：课程名、当前助教、运行状态。
2. 主卡片：今日建议。
   - 建议动作：例如“复习 RAG 检索策略并完成 3 道小测”。
   - 推荐原因：来自画像、最近错题、课程进度。
   - 主按钮：开始今天的学习。
3. 右侧或下方：快捷动作。
   - 继续上次问题。
   - 生成一道练习。
   - 看图解。
   - 复盘错题。
   - 生成语音讲解。
4. 辅助信息：最近学习记录、资料引用、待确认记忆。

### 7.3 视觉风格

遵循 `DESIGN-notion.md` 的 SparkWeave Product Adaptation：

- 用白色 canvas 和浅灰 surface，保持工作台感。
- 卡片边框用 hairline，减少大面积阴影。
- 按钮和卡片圆角控制在 8px 内。
- 紫色只用于主行动和少数强调，不做大面积背景。
- 采用 pastel note-like callout 表达不同资源类型。
- 不在默认页面展示 JSON、route、runtime、WS、tool_call 等术语。

## 8. 核心用户流程

### 8.1 首次创建助教

当前表单改为三步向导：

1. 选择课程。
   - 大模型与智能学习系统。
   - 机器学习基础。
   - 自定义课程。
2. 选择助教风格。
   - 耐心讲解型。
   - 追问启发型。
   - 刷题训练型。
   - 项目导师型。
3. 绑定资料和画像。
   - 知识库。
   - 当前学习画像。
   - 课程模板。
   - 语音/OCR/TTS 能力开关。

完成后自动创建 SparkBot，并写入合适的 `SOUL.md`、`USER.md`、`TOOLS.md`、`AGENTS.md`。

### 8.2 继续今天的学习

流程：

1. 学生进入 AI 助教中心。
2. 系统读取学习画像、最近练习、课程进度和知识库状态。
3. 给出今日一步建议。
4. 学生点击“开始今天的学习”。
5. 助教先说明学习目标，再进入讲解或练习。
6. 完成后写入学习记录，并更新下一步建议。

### 8.3 即时答疑

流程：

1. 学生输入文字、语音或上传截图。
2. 系统识别当前问题类型：概念解释、解题、代码、图解、资料检索、复盘。
3. 协调智能体选择能力：RAG、deep_solve、deep_question、visualize、math_animator 或普通 chat。
4. 助教返回答案。
5. 答案附带来源、协作路线、下一步动作和反馈按钮。

### 8.4 练习与评估闭环

流程：

1. 助教基于薄弱点生成小测。
2. 学生作答。
3. 系统判断正确性、错因和知识点。
4. 生成解释和同类变式题。
5. 写入学习效果事件。
6. 更新掌握度和下一步资源推荐。

### 8.5 多模态讲解

流程：

1. 学生点击“看图解”或“生成短讲解”。
2. 助教生成讲解脚本、图解结构和语音内容。
3. 使用可视化能力生成图解，使用讯飞 TTS 生成音频。
4. 必要时生成短视频或视频脚本。
5. 产物进入“资料与产物”区域，可下载或复习。

## 9. 功能规划

### 9.1 我的助教

必须有：

- 当前助教卡。
- 今日学习建议。
- 推荐原因。
- 一个主行动按钮。
- 最近学习活动。
- 常用快捷动作。

建议文案：

- “今天先补 RAG 检索策略。”
- “依据：你最近两次在召回率和重排题上出错，课程第 2 章还未完成。”
- “完成后我会更新你的知识点掌握度。”

### 9.2 助教对话

对话区优化：

- 输入框上方显示当前课程、知识库和画像状态。
- 输入框附近提供快捷动作，而不是只靠自然语言 prompt。
- 过程状态翻译为学习语言。
- 答案底部展示：
  - 来源依据。
  - 智能体协作路线。
  - 下一步动作。
  - 反馈按钮。

反馈按钮：

- 有帮助。
- 太难了。
- 想要例子。
- 这个回答不准确。
- 记入学习记录。

### 9.3 助教能力

将当前 capability 矩阵改为学习任务卡：

| 任务卡 | 对应能力 |
| --- | --- |
| 讲懂一个概念 | chat + rag + reason |
| 分步解一道题 | deep_solve |
| 生成一组练习 | deep_question |
| 画一张图解 | visualize |
| 生成短视频讲解 | math_animator 或视频脚本 + TTS |
| 查找拓展资料 | deep_research + web_search + paper_search |
| 复盘错题 | learning_effect + deep_question |
| 制定本周计划 | learner_profile + guide |

### 9.4 资料与产物

新增“助教产物”视角，不直接把所有内容暴露为 workspace 文件。

产物类型：

- 讲解卡。
- 练习卡。
- 错题复盘。
- 图解。
- 音频讲解。
- 视频讲解或视频脚本。
- 课程摘要。
- 项目 rubric。
- 导出 Markdown。

当前实现已用 history 和 workspace files 承载可演示产物，并在资料与产物区形成可复用的课程包、讲解卡、练习卡和多模态资源入口。

### 9.5 学习记录

学习记录不做成“成绩墙”，而是回答：

- 最近系统根据什么判断我薄弱。
- 我完成了哪些动作。
- 哪些知识点掌握度变化了。
- 下一步为什么是这个。

需复用现有学习画像和学习效果设计：

- 证据账本。
- 掌握度。
- 错因闭环。
- 资源偏好。
- 下一步动作。

### 9.6 渠道与高级

保留但后置：

- 外部渠道配置。
- Soul 模板。
- 工作区文件编辑。
- tools JSON。
- runtime JSON。
- heartbeat。
- cron。
- nano-team。

在比赛演示中默认不展开，除非需要展示工程能力。

### 10.2 后端实施

  - 方案 B：前端不再提示创建任意 `NOTES.md`，改为固定可编辑文件。
  - 推荐方案 A，因为助教学习笔记确实需要一个用户可理解的文件。

- 增加可展示的来源摘要。
  - 若本轮调用 RAG，前端展示 sources。
  - 若无来源，明确显示“未使用课程资料”或“来自通用模型知识”。

### 10.3 智能体编排

助教中心应把多智能体协作显性化，但不要让用户看到技术细节。

推荐展示格式：

```text
学习画像 -> 课程资料检索 -> 讲解智能体 -> 练习智能体 -> 学习记录
```

映射到现有能力：

- 学习画像：learner profile。
- 课程资料检索：RAG 或星火知识库。
- 讲解智能体：chat、reason、deep_solve。
- 练习智能体：deep_question。
- 图解智能体：visualize。
- 评估智能体：learning_effect。
- 长期陪伴：SparkBot memory、heartbeat、cron。

### 10.4 讯飞工具接入

| 能力 | 接入方式 | 演示价值 |

| 讯飞语音合成 | 讲解音频、短视频配音 | 多模态输出 |
| 讯飞 OCR/公式识别 | 图片题目、讲义截图识别 | 拍照问答 |
| 讯飞 embedding | 知识库向量化 | 私域资源检索 |

## 11. 分阶段计划

### P0：计划与主线冻结

目标：明确产品主线和演示课程。

- 演示课程 `ai_learning_agents_systems` 大纲。
- 助教首页线框草图。

验收：

- 能用一句话说明 AI 助教中心价值。
- 能把功能映射到赛题每个要求。

### P1：助教中心体验重组

目标：不大改后端，先让页面像学习产品。

任务：

- 将页面标题、hero、tab、卡片文案改为“AI 助教中心”。
- 新增“今日学习建议”工作台。
- 新增快捷动作按钮。
- 将文本命令包装为按钮或菜单。
- 修复 `NOTES.md` 前后端不一致。
- 将过程状态翻译为学习语言。
- 高级配置默认折叠。

验收：

- 学生进入页面 5 秒内能知道下一步做什么。
- 不进入高级区也能完成一次答疑、练习生成和复盘。
- 页面不出现明显技术噪声。

### P2：个性化学习闭环

目标：让助教真正使用画像和评估。

任务：

- 接入学习画像摘要：当前课程、薄弱点、偏好资源。
- 接入学习效果 next actions。
- 助教回答后显示反馈按钮。
- 反馈写入证据账本或学习效果事件。
- 练习结果回写掌握度和错因。
- 展示“为什么推荐”。

验收：

- 今日建议能基于画像或最近练习解释。
- 反馈后能在学习记录中看到变化。
- 回答底部至少给出 2 到 3 个下一步动作。

### P3：资料来源与多模态资源

目标：展示智能辅导加分项。

任务：

- RAG 来源在回答中可视化。
- 图解生成入口接入 visualize。
- 语音提问接入讯飞语音听写。
- 语音讲解接入讯飞 TTS。
- 图片题目或讲义截图接入 OCR/公式识别。
- 产物进入“资料与产物”区域。

验收：

- 同一问题可以生成文字解释、图解和语音讲解。
- 来源可点击或可追溯。
- 产物可复用，不只停留在聊天气泡中。

### P4：比赛演示打磨

目标：形成 7 分钟内稳定可复现演示。

任务：

- 固定演示数据：课程、画像、知识库、样例错题。
- 固定演示助教模板。
- 增加一键 demo seed 脚本或说明。
- 整理 PPT 截图素材。
- 录制演示视频脚本。
- 撰写 AI Coding 工具说明。

验收：

- 从打开页面到完成闭环不超过 7 分钟。
- 演示覆盖画像、资源生成、路径推荐、智能辅导、效果评估。
- 明确出现科大讯飞工具接入说明或演示。

## 13. 验收指标

### 13.1 用户体验指标

- 首屏有且只有一个主行动。
- 默认视图不出现 JSON、runtime、channel、tool_call 等术语。
- 常用学习任务最多两次点击可触达。
- 每次回答后至少提供一个可执行下一步。
- 学习建议必须有原因。
- 重要资源必须可追溯来源。

### 13.2 技术指标

- SparkBot 创建、启动、停止、聊天功能保持可用。
- 工作区文件保存不破坏现有 allowlist 测试。
- 反馈事件能被学习画像或学习效果模块消费。
- RAG 资料来源能在前端稳定展示。
- 讯飞能力配置缺失时有降级提示，不阻断普通文字助教。

### 13.3 比赛指标

- 至少一门完整高校课程资料可用。
- 至少一次多智能体协作路线可展示。
- 至少一次多模态输入或输出可展示。
- 至少一次学习效果评估闭环可展示。
- 明确说明使用了哪些科大讯飞能力。

## 14. 风险与应对

| 风险 | 应对 |
| --- | --- |
| 前端一次性重构过大 | 先在现有页面内新增工作台，稳定后拆组件 |
| SparkBot 与 unified runtime 能力重复 | 当前阶段用 SparkBot 承载长期助教，用 capability 做任务能力 |
| 讯飞工具密钥或额度不稳定 | 做 provider 配置检测和降级文案，演示前固定环境 |
| RAG 来源缺失导致回答不可追溯 | 未使用资料时显式提示，并引导选择课程资料 |
| 学习画像误判 | 增加“确认/编辑/忘记”机制，偏好只由强证据强化 |
| 高级能力吓到普通学生 | 高级配置默认折叠，入口命名为“渠道与高级” |
| 产物散落在聊天历史 | 第一阶段提供产物筛选，第二阶段建立 artifact store |

## 16. 总结

AI 助教中心的关键不是再增加一个聊天页面，而是把 SparkWeave 已有的学习画像、RAG、多智能体、资源生成、讯飞多模态能力和学习效果评估组织成一条学生能理解的路径。

当前阶段已经解决“学生进来不知道做什么”的问题：首屏能给出今日建议、原因和明确行动，SparkBot 的复杂能力已经从后台技术变成可感知的学习体验。来源、反馈、产物和多模态闭环也已补齐，形成适合比赛展示的完整智能助教方案。
