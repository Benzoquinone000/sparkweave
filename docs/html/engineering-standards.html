软件工程规范

本篇面向评委说明 SparkWeave 的工程边界。一个参赛作品不只要能运行，还要让评委能看清源码、课程、配置、数据和验证结果各自在哪里，哪些内容可以公开提交，哪些内容必须留在本地运行环境。

基本原则

SparkWeave 的工程规范围绕四件事：

原则 含义
可运行 提交包应能通过 Docker Compose 启动前端、后端和 Milvus
可核验 课程模板、API 合同、前端构建和资料问答都有对应验证入口
可追溯 资料问答展示来源，学习画像保留证据，配置状态可以检查
有边界 不把真实密钥、真实学生数据和本机运行数据混入提交包

这四点比“功能数量多”更重要。评审时最怕的是页面很多、能力很多，但无法判断哪一部分真正运行、哪一部分只是静态材料。

目录职责

目录或文件 职责
sparkweave/ 后端 API、业务服务、多智能体运行时、RAG、学习画像和讯飞工具
web/ React 前端工作台和页面截图
data/course_templates/ 随项目提交的完整课程模板
data/user/ 运行后生成的学习记录、画像、设置和笔记
data/knowledge_bases/ 资料页生成的知识库文件和索引信息
data/milvus/ Milvus 本地运行数据
scripts/ 项目验证、课程模板检查、API 合同检查和资料问答验收
tests/ 后端、服务层、RAG、API 和验证入口的自动测试
docs/ 面向评委的设计、配置、数据和验证说明
.env.example 可公开的配置模板
.env 本机真实密钥，不提交

目录职责清楚后，评委可以快速判断：课程在哪里、页面在哪里、后端在哪里、运行数据为什么不一定随提交包出现。

前端体验边界

SparkWeave 前端遵循一个约束：优先服务学习用户，而不是把所有能力铺成展示菜单。

设计边界 要求
主入口 默认保留学习、资料、记录、设置
更多工具 问问题、练习、课程助教、图像解题等放在学习工具或高级工具中
页面主动作 每个页面先给出一个明确动作，例如继续学习、上传资料、查看建议
空状态 不能只写“暂无数据”，要告诉用户下一步做什么
技术名词 能用学习动作表达时，不用资料检索、智能体、运行时这类技术词做入口

这样处理后，评委看到的是一条完整学习流程，而不是一个功能展台。

后端能力边界

后端能力按“页面能用、过程可见、数据能落地”组织。

能力 工程要求
学习路线 读取课程模板，生成任务、资源建议和学习报告
学习画像 学习行为进入证据记录，再生成画像和下一步建议
资料问答 上传资料后建立索引，回答时展示来源
多智能体资源生成 长过程返回阶段进度，让前端展示处理过程
讯飞工具链 设置页可配置，系统状态可检测，调用结果可识别
课程助教 与课程文件、历史记录和提醒任务保持边界

新增能力时，应先确认它是否进入这些学习流程。如果只是为了展示技术名词，优先放到更多工具或扩展入口，不打断主路径。

数据与密钥边界

正式提交应包含：

  • 源码、前端工程、Docker 配置和依赖说明。
  • data/course_templates/ 中的完整课程模板。
  • .env.example 等不含密钥的配置示例。
  • 面向评委的文档和当前界面截图。
  • 可公开展示且来源清楚的少量示例资料。

正式提交不应包含：

  • .env、真实 API Key、账号 Token、私有服务地址。
  • 真实学生姓名、聊天记录、学习记录、语音材料和个人画像。
  • 未获授权的教材、课件、论文全文或课程资料。
  • data/milvus/、data/user/ 这类运行后生成的数据。
  • node_modules/、.venv/、构建产物、日志和临时压缩包。

如果确实需要展示已有学习记录，应使用脱敏样例，并在文档中说明它是示例数据。

问题 对应文档

数据哪些能提交 data-storage-guide.md

文档应尽量用真实页面、真实路径和真实验证方式说话。不要把未实现计划写成已完成能力，也不要用大段口号替代系统行为。

质量门禁

项目提供几类验证入口：

验证 命令
项目结构、文档索引、提交边界 python scripts/verify_project.py --profile quick
课程模板 python scripts/check_course_templates.py
前后端 API 合同 python scripts/check_web_api_contract.py
发布前敏感内容 python scripts/check_release_safety.py
Python 编译 python -m compileall -q sparkweave sparkweave_cli scripts
后端测试 python -m pytest tests
前端验证 npm run verify（在 web/ 目录）

这些检查不替代人工演示，但能证明提交包的基本结构、接口、课程模板和前端构建没有明显断裂。

赛题交付口径

面向 A3 赛题，SparkWeave 的交付口径应保持一致：

  • 主课程使用 深度学习，避免视频录制或核验时频繁切换主题。
  • 展示按“画像 -> 路径 -> 资料 -> 辅导 -> 评估 -> 讯飞工具”组织。
  • 截图、视频、PPT 和现场系统使用同一套页面和课程数据。
  • 讯飞工具配置在设置页展示，未配置的服务要如实显示状态。
  • AI Coding 工具只作为辅助说明，不把责任推给工具。

边界说明

工程规范不是为了显得项目复杂，而是为了让交付更可信。评委拿到 SparkWeave 后，应当能看清三件事：源码和课程模板足以运行系统；运行后产生的数据有明确位置；密钥、个人数据和本机产物没有混入提交包。做到这三点，作品功能和工程交付才是同一件事。
