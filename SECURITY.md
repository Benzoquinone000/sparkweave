# SparkWeave 安全策略

范围：说明当前仓库的安全边界、敏感信息处理和漏洞报告方式。本文档不承诺托管服务 SLA；当前项目默认本地单用户运行，生产化或多用户部署前必须做额外隔离与权限设计。

## 支持范围

| 范围 | 状态 |
| --- | --- |
| `main` 分支 | 接受安全修复和发布前检查 |
| 本地 Docker Compose 运行 | 当前主要支持场景 |
| 多用户公网部署 | 需要部署方自行补齐身份认证、隔离、审计和密钥管理 |

## 敏感信息

不要提交：

- `.env`、真实 API Key、OAuth token、账号 JSON、Cookie、录屏中的明文密钥。
- `data/user/`、`data/knowledge_bases/`、`data/milvus/`、`data/memory/` 中的真实用户数据。
- 本地下载目录、临时凭证压缩包、供应商导入文件。

提交前运行：

```powershell
python scripts/check_release_safety.py
python scripts/check_project_standards.py
```

## 漏洞报告

如果发现安全问题，请优先使用 GitHub 的私密漏洞报告或安全公告入口。不要在公开 issue 中粘贴可利用细节、密钥、真实用户数据或完整攻击载荷。

报告时建议包含：

- 受影响的代码路径、接口或命令。
- 复现步骤和预期影响。
- 是否需要真实外部服务、真实密钥或特定本地配置。
- 建议修复方向或最小化验证用例。

## 安全开发要求

- 文件上传、压缩包解包和静态文件输出必须校验路径边界，防止目录穿越。
- 命令执行、代码执行和工具调用必须显式限制输入大小、工作目录和输出。
- 外部供应商调用失败时应返回结构化错误或离线替补，不把内部异常和密钥打印给用户。
- API 新增字段要避免回传密钥；前端设置页只能展示必要的掩码或状态。
- 新增安全敏感能力必须覆盖 `tests/api/`、`tests/services/`、`tests/core/` 或 `tests/scripts/` 中的相关测试。
